## 1) Что за задача и что означает `score`

### 1.1 Что есть в данных
Данные приведены к панели **driver-month**:

- **driver** = водитель (у каждого есть `driver_id`)
- **month** = месяц (например `2024-06-01`)
- **features** = численные признаки (агрегаты по истории)
- **target** = 0/1 (“плохое событие” произойдёт в будущем горизонте или нет)

То есть одна строка = “вот такой водитель был в таком-то месяце”.

### 1.2 Что делает модель
Модель обучается и для каждой строки выдаёт число `score` (обычно от 0 до 1):

- `score` ближе к 1 → модель считает случай более рискованным
- `score` ближе к 0 → менее рискованным

**Важно:** в нашем проекте `score` используется в первую очередь как **ранжирование**.  
То есть мы сортируем строки/водителей по score и берём top‑K.

---

## 2) Почему у нас два уровня оценки: driver-month и driver-level

### 2.1 Driver-month (водитель × месяц)
Мы оцениваем, как модель ранжирует “опасные месяцы”. Это удобно, потому что:
- строк больше → обучение стабильнее
- видно поведение по времени

### 2.2 Driver-level (водитель)
На практике часто нужен список **водителей**, а не список “водитель‑месяц”.
Поэтому мы берём все месяцы одного водителя и агрегируем его score → получаем один score на водителя.

И дальше можно сделать:
- “топ‑30 самых рискованных водителей”
- “топ‑100”
- и т.д.

---

## 3) Что такое `agg` (агрегации driver-level) простыми словами

У одного водителя за тестовый период несколько месяцев → несколько score.  
Чтобы получить один итоговый score, мы используем агрегации:

- `last` — берём score **последнего месяца**  
  Смысл: “риск прямо сейчас”.

- `max` — берём **максимальный** score за окно  
  Смысл: “если хоть раз было очень плохо — поднимаем вверх”.

- `mean` — берём **средний** score за окно  
  Смысл: “стабильный средний риск”.

- `topq_mean` — берём среднее по “верхним” месяцам (например по топ‑20% месяцев по score)  
  Смысл: компромисс: учитываем пики, но не один-единственный случайный всплеск.

Почему это важно: разные `agg` реально дают разные рейтинги водителей и разные метрики.

## 4) Какими метриками мы пользуемся (на пальцах)

Мы используем два типа метрик:

### 4.1 “Общие” метрики ранжирования
- **ROC-AUC** — насколько хорошо модель в среднем ставит positives выше negatives
- **AP (Average Precision)** — похожая идея, но больше “про верх списка” (часто лучше при дисбалансе)

### 4.2 Top‑K метрики (самые понятные)
Берём top‑K объектов по score и считаем:
- `tp@K` — сколько “плохих” мы нашли в топе
- `fn@K` — сколько “плохих” пропустили (не попали в топ)
- `recall@K` — доля найденных плохих
- `precision@K` — насколько “чистый” топ (сколько из топа реально плохие)
- `miss_rate@K` — доля пропущенных плохих (это `1 - recall@K`)

Подробно и спокойно это объяснено в файле:
- `stage_2/about_metrics.md`

## 5) Где что лежит (карта репозитория)

### 5.1 Главное: stage_2
- `stage_2/data.py`  
  Сборка датасета и временной split train/test.

- `stage_2/models/`  
  Скрипты обучения моделей. Важная идея: **каждая модель — отдельный модуль**, запускается через `python -m`.

  Примеры:
  - `stage_2/models/xgb.py` — XGBoost
  - `stage_2/models/hgb.py` — HistGradientBoosting (sklearn)
  - `stage_2/models/extratrees.py` — ExtraTrees
  - `stage_2/models/easyensemble.py` — EasyEnsemble (imbalanced-learn)
  - `stage_2/models/logreg.py` — Logistic Regression
  - `stage_2/models/catbst.py` — CatBoost (файл назван так, чтобы не конфликтовать с пакетом)

- `stage_2/metrics/`  
  Код метрик:
  - `driver_month_metrics.py` — метрики на уровне driver-month
  - `driver_metrics.py` — агрегирование в driver-level и метрики на уровне водителей

- `stage_2/output/`  
  Папка с артефактами запусков (CSV).

- `stage_2/summarize_metrics.py`  
  Скрипт, который берёт все результаты моделей из `stage_2/output/` и печатает сравнительные таблицы.

### 5.2 Документы (без сухих формул)
- `stage_2/about_models.md` — какие модели выбраны, как они работают, плюсы/минусы
- `stage_2/about_metrics.md` — какие метрики и как их понимать
- `stage_2/about_results.md` — как читать наши результаты и какие выводы из них следуют (без копирования огромных таблиц)

## 6) Как всё запустить (пошагово)

### 6.1 Виртуальное окружение
Из корня репозитория:

```bash
python -m venv .venv
source .venv/bin/activate
pip install -U pip
```

### 6.2 Установка зависимостей
Если есть `requirements.txt`:
```bash
pip install -r requirements.txt
```

Если requirements нет, минимальный набор:
```bash
pip install numpy pandas scikit-learn xgboost catboost imbalanced-learn
```

### 6.3 Запуск моделей
Примеры:

```bash
python -m stage_2.models.extratrees
python -m stage_2.models.easyensemble
python -m stage_2.models.xgb
python -m stage_2.models.hgb
python -m stage_2.models.logreg
python -m stage_2.models.catbst
```

Что происходит при запуске модели:
1) строится панель `driver-month`
2) делается split по времени (train — прошлое, test — будущее)
3) обучается модель
4) считаются метрики
5) сохраняются CSV в `stage_2/output/`

### 6.4 Сравнение всех моделей одной командой
После того как прогонил хотя бы 2–3 модели:

```bash
python -m stage_2.summarize_metrics
```

Он:
- читает `stage_2/output/*_driver_month_scored.csv`
- печатает сравнение в формате, который влезает в экран
- сохраняет “сводные CSV”:
  - `stage_2/output/summary_driver_month_metrics.csv`
  - `stage_2/output/summary_driver_metrics.csv`

## 7) Какие файлы появляются в `stage_2/output/` и что они значат

После запуска модели `<model>` обычно появляются:

### 7.1 `*_driver_month_scored.csv`
Пример:
- `extratrees_driver_month_scored.csv`

Что внутри:
- `driver_id`
- `month`
- `y_true` (0/1)
- `score` (предсказание модели)

Это **главный универсальный артефакт**, потому что из него можно:
- пересчитать любые метрики,
- делать новые агрегации driver-level,
- строить графики,
- делать ансамбли (если захотеть).

### 7.2 `*_drivers_ranked_<agg>.csv`
Примеры:
- `extratrees_drivers_ranked_last.csv`
- `extratrees_drivers_ranked_max.csv`
- `extratrees_drivers_ranked_mean.csv`
- `extratrees_drivers_ranked_topq_mean.csv`

Что внутри:
- по сути “таблица водителей” с одним score на водителя по выбранной агрегации,
- обычно уже отсортирована по score.

### 7.3 (иногда) `*_feature_importance.csv`
Пример:
- `extratrees_feature_importance.csv`

Это вспомогательная таблица: какие признаки модель считает важными.

## 8) Частые проблемы (и быстрые решения)

### 8.1 “ModuleNotFoundError: imblearn”
Поставь:
```bash
pip install imbalanced-learn
```

### 8.2 CatBoost/XGBoost не ставятся
Иногда нужно обновить pip и поставить отдельно:
```bash
pip install -U pip
pip install xgboost catboost
```

### 8.3 Не запускается `python -m stage_2....`
Проверь:
- ты в корне репозитория
- активирована `.venv`
- папка `stage_2/` реально существует

### 8.4 Почему обучение “слишком быстро”
Потому что данных немного (тысячи строк, десятки признаков), а модели оптимизированы.  
Долго становится при кросс‑валидации и переборе параметров.

## 9) Соглашения по именам файлов (важно для Python)
Не называй файлы как библиотеки, иначе Python может импортировать твой файл вместо пакета:

Плохо:
- `catboost.py`
- `xgboost.py`

Хорошо:
- `catbst.py`
- `xgb.py`
